# Performance and Stress Testing

## Purpose

The performance runner validates the existing two-step ingestion workflow without adding test-only behavior to the API:

1. Generate a deterministic EDG-compatible CSV workload.
2. Call `POST /api/upload`.
3. Capture the returned `ReportUID`.
4. Call `POST /api/ingestion/{reportUid}`.
5. Capture counts, retries, batch progress, Parquet details, HTTP status, and elapsed time.
6. Generate JSON, Markdown, and HTML reports from measured results.

The runner does not manufacture performance numbers. Every value in the report is calculated from the generated input file, API responses, and measured wall-clock duration.

## Test levels

### Unit tests

Unit tests run during every pull request. They use xUnit and do not call AWS or SMB resources.

```bash
dotnet test RemediationTool.sln \
  --configuration Release \
  --settings coverage.runsettings \
  --collect:"XPlat Code Coverage"
```

### Baseline load test

A baseline test proves the workflow with a known workload and low concurrency.

```bash
dotnet run \
  --project tests/RemediationTool.PerformanceTests \
  --configuration Release -- \
  --base-url https://remediation-api.example \
  --records 10000 \
  --concurrency 1 \
  --invalid-percent 0 \
  --scenario ingestion-baseline \
  --output artifacts/performance/baseline-10000
```

Use the environment variable `REMEDIATION_BEARER_TOKEN` for authentication. Do not place access tokens in command history or workflow arguments.

### Stress test

A stress test increases volume or concurrent jobs until throughput degrades, retries increase, throttling appears, or the application reaches an agreed resource limit. A single successful execution is a load test, not a complete stress test.

Recommended progression:

| Scenario | Records per job | Concurrency | Purpose |
|---|---:|---:|---|
| Smoke | 1,000 | 1 | Verify environment and credentials |
| Baseline | 10,000 | 1 | Establish normal throughput |
| Medium | 100,000 | 1 | Validate batching and Parquet |
| High volume | 500,000 | 1 | Validate memory and long-running behavior |
| Concurrent | 100,000 | 2–5 | Identify shared-resource contention |
| Rejection | 100,000 | 1 | Validate rejected-row throughput |
| Retry | 100,000 | 1 | Validate throttling and retry behavior |
| Requirement | 1,000,000+ | 1 | Validate million-record ingestion |

Increase only one primary variable at a time when establishing a baseline. This makes performance changes explainable.

## Generated artifacts

Each execution creates:

```text
artifacts/performance/
├── stress-results.json
├── STRESS_TEST_RESULTS.md
├── STRESS_TEST_RESULTS.html
├── STRESS_TEST_RESULTS.pdf       # generated by GitHub Actions when available
├── upload-response-01.json
├── ingestion-response-01.json
└── ...
```

The generated CSV files are removed after execution by default to avoid retaining large temporary data. Set `REMEDIATION_KEEP_INPUT=true` only when the source workload is required for investigation.

## Pass criteria

The runner marks a scenario as passed when:

- every job receives a successful HTTP response;
- no job ends in `Failed` or `Unknown` status;
- all generated input records are included in the ingestion result;
- expected validation rejections do not become infrastructure failures.

A partial-success response is valid when invalid rows were intentionally generated. Rejected rows remain visible in the report and contribute to the rejection rate.

## GitHub Actions

The `Quality and Performance` workflow provides two independent paths:

- Pull request and main-branch runs build the solution, execute unit tests, collect Cobertura coverage, and upload test artifacts.
- Manual workflow runs execute the performance runner against a supplied API URL and upload the generated report bundle.

Configure the `performance-test` GitHub environment with:

- `REMEDIATION_BEARER_TOKEN` secret when authentication is enabled;
- environment protection or approval rules for shared test environments;
- a self-hosted runner when the target API is reachable only from the internal network.

## Environment safety

Run performance tests only against isolated resources:

- dedicated DynamoDB tables or test partitions;
- dedicated S3 bucket or prefix;
- non-production credentials;
- explicit cleanup and retention rules;
- CloudWatch or Grafana monitoring enabled during the execution.

Never run the million-record or concurrency scenarios against production without written approval and a rollback or stop plan.

## Bottleneck analysis

The report compares upload and ingestion API duration. Use the existing structured `INGESTION_STAGE_COMPLETE` application logs to identify internal cost in:

- source download;
- parsing and validation;
- rejected-row persistence;
- Parquet or staging preparation;
- target DynamoDB persistence;
- processing-summary creation;
- staging cleanup.

Correlate the report run ID, `ReportUID`, Git commit, API logs, DynamoDB throttling, CPU, memory, and garbage collection before making concurrency changes.
